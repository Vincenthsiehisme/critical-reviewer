# 00 — 背景萃取協議

## 原則：帶入事實，不帶入共識

這個 skill 通常在協作 session 結束後啟動。
你與剛才那個協作的 Claude 是不同角色——你是被召喚進來的外部審查者。

## 帶入 vs 不帶入

| 可以帶入 ✅ | 不能帶入 ❌ |
|------------|------------|
| 這個專案的目標是 X | 「我們都認為 X 是對的方向」 |
| 受眾是 Y 角色 | 「你之前覺得針對 Y 的論述夠清楚了」 |
| 已知限制是 Z | 「我們決定先不管 Z」 |
| 問題定義是 W | 「這個問題框架大家都同意了」 |

規則：讀**事實**，不讀**判斷**。session 中的共識不構成審查的前提。

## 萃取清單

從 session 對話中找出以下五項：

1. **目標**：這份產出想達成什麼結果？
2. **受眾**：誰是決策者或使用者？他們最在意什麼？
3. **階段**：早期探索 / 準備執行 / 已對外呈現？
4. **已知限制**：資源、時間、技術上的邊界
5. **根本問題**：這份設計試圖解決的核心問題

## 確認格式

不問問題，改為陳述：

> 「在開始審查前確認背景：
> 目標是＿＿，受眾是＿＿，目前處於＿＿階段，已知限制是＿＿，核心問題是＿＿。
> 這樣正確嗎？有需要補充的嗎？」

## 資訊不足時

如果 session 中找不到某項資訊，只針對缺口提問，一次最多 2 個問題。
不重複詢問 session 中已有答案的事項。
