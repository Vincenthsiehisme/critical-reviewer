# critical-reviewer

> 以麥肯錫框架召集外部顧問小組，對概念設計與規劃產出進行結構化批判性審查的 Claude skill。

---

## 這個 Skill 做什麼

在 AI 協作完一個 session 後，召喚一組獨立的外部顧問——他們沒有看過你們剛才的對話，從第一性原理重新審視產出。

每位顧問：
- 持有特定的麥肯錫框架工具箱
- 發言前執行獨立搜尋，用外部現實校準判斷
- 不看其他顧問的發言，獨立作業
- 只在找到問題時才發言——沉默是主動判斷，不是找不到問題

整合段交叉比對三個獨立調查，找出「獨立收斂」（多個顧問各自指向同一問題）作為最高優先級訊號。

---

## 顧問角色

| 顧問 | 核心問句 | 框架工具箱 | 搜尋策略 |
|------|---------|-----------|---------|
| 🔷 策略顧問 | 你在解決對的問題嗎？值得嗎？ | MECE、Issue Tree、Prioritization Matrix | 市場現況、競品做法 |
| 🔶 使用者研究員 | 你真的了解你的用戶嗎？ | 假設驅動、So What / Why So | 用戶行為研究、同類產品反饋 |
| 🔴 風險顧問 | 最壞的情況是什麼？ | MECE（遺漏視角）、Issue Tree（失敗路徑） | 同類方案失敗案例、常見執行陷阱 |

---

## 觸發方式

在 Claude 對話中說：
- 「幫我審查」「批判這份」「挑毛病」
- 「devil's advocate」「找盲點」「第二意見」「顧問怎麼看」
- 貼上概念文件、設計稿、規劃草案
- 「這樣 ok 嗎」「哪裡有問題」

---

## 執行流程

```
Phase 0  背景萃取      從 session 讀取事實背景，不繼承共識與判斷
Phase 1  快速掃描      診斷框架缺口，決定召集哪些顧問
Phase 2  顧問召集      依掃描結果，只載入被召集顧問的角色檔案
Phase 3  獨立調查      各顧問搜尋 → 套用框架 → 獨立發言（互相隔離）
Phase 4  交叉比對      整合三方調查，標示獨立收斂、單一發現、顧問張力
```

## 設計原則

**帶入事實，不帶入共識**
顧問從 session 讀取目標、受眾、限制、問題定義，但不繼承「我們之前說好的方向」。

**框架紀律優於自然語言描述**
每個問題必須標示框架名稱，並說明框架在這份產出上實際揭示了什麼。把框架名稱拿掉後分析仍成立，代表框架沒有真正被使用。

**搜尋用來找差距，不用來找背書**
顧問搜尋的目的是找「外部現實與產出假設的矛盾」，找到支持產出的資訊不引用。

**獨立收斂是最高優先級訊號**
多個顧問從不同資訊來源、不同框架，獨立指向同一個問題——這個訊號強度高於任何單一顧問的發現。
