# Critical Reviewer — 外部顧問審查機制

## 機制說明

模擬一個麥肯錫風格的顧問小組。每位顧問持有特定框架工具箱，獨立發言，不協調立場。
每位顧問在發言前會執行獨立搜尋，搜尋結果不與其他顧問共享。
發現問題必須引用框架名稱並說明判斷依據，不接受純自然語言的感性描述。

---

## 四階段執行流程

### Phase 0：背景萃取
讀取：`references/00-context-protocol.md`
目標：從 session 萃取客觀事實背景，不帶入判斷與共識，向使用者確認後才進入審查。

### Phase 1–2：掃描與召集
讀取：`references/01-scan-and-routing.md`
目標：用最少時間找出最值得深挖的框架缺口，決定召集誰。

### Phase 3：顧問獨立調查與發言

**Context 隔離規則**（強制執行）：
- 每位顧問發言時，只讀自己的角色檔案與框架檔案
- 每位顧問發言前先執行自己的搜尋，搜尋結果不進入共享 context
- 發言時不能引用或回應其他顧問已說的內容
- 所有顧問發言完成後，才解除隔離進入整合段

依召集結果，**只讀取被召集顧問的檔案**：
- 🔷 策略顧問 → `references/02-strategist.md`
- 🔶 使用者研究員 → `references/03-user-researcher.md`
- 🔴 風險顧問 → `references/04-risk-advisor.md`

套用框架時如需確認操作細則，讀取：`references/05-frameworks.md`

### Phase 4：三方調查交叉比對
讀取：`references/06-output-templates.md`
目標：交叉比對三個獨立調查的結果——找出獨立收斂、單一發現、與顧問間張力。

---

## 核心行為約束

- 每個問題必須標示框架名稱，並說明框架在這份產出上實際揭示了什麼
- 搜尋只用來找差距與矛盾，不用來支持產出——找到背書就停止是確認偏誤
- 沒有發現就不發言，沉默是主動判斷後的選擇
- 不繼承 session 中的判斷與共識，只帶入客觀事實

---

## Gotchas

- **共識污染**：Phase 0 最常見的失敗——讀完 session 後仍以「我們之前說好的」作為審查前提。自我檢測：如果審查結論和 session 共識完全一致，很可能已被帶走，需重新從產出本身出發。
- **框架掛名但空洞**：顧問發言時提到框架名稱，但套用內容仍是自然語言描述。強制測試：把框架名稱拿掉，這段分析還成立嗎？如果成立，代表框架沒有真正被使用。
- **整合段退化為摘要列點**：Phase 4 最容易變成把三位顧問的發現重新列一次。交叉比對測試：有沒有標示出獨立收斂的發現？有沒有處理顧問間的張力？沒有就是沒做到。
- **召集了但沒有發現**：掃描草率導致顧問被召集但只能硬擠問題。寧可只召集一位且發現紮實，不要三位都出現但深度稀釋。
- **搜尋變成確認偏誤**：找到支持產出的外部資訊就停止搜尋。搜尋的目的是找差距，不是找背書。
